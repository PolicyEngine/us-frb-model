# Model-consistent expectations (MCE): feasibility assessment

Status: **assessment only** — nothing here is implemented. `Frbus(..., mce=...)`
currently raises `NotImplementedError` (src/frbus/frbus.py) and the parser
already collects the names of variables carrying an `<mce_equation>`
(`ModelSpec.mce_vars`, src/frbus/parse.py).

## What MCE is in FRB/US terms

Under VAR expectations (what we implement) every equation is backward-looking,
so the model solves period by period. Under MCE, a subset of equations is
replaced by their `<mce_equation>` variants, which contain **leads** —
references like `zpic58(+1)` to future model outcomes. Expectations then equal
the model's own future solution, which requires solving all simulation periods
**simultaneously**. The vendor supports four variants (`mcap`, `wp`,
`mcap+wp`, `all` — `pyfrbus/constants.py`), selecting which sectors (asset
pricing, wage-price, or all) become forward-looking.

## How the vendor does it (vendor/pyfrbus_package)

The vendored pyfrbus is the working reference; its MCE machinery is:

1. **Equation swap-in** (`frbus.py:70-80`, `xml_model.mce_from_xml`): for the
   chosen variant, replace each MCE variable's standard equation with its
   `mce_equation` and merge the MCE coefficient set.
2. **Stacked-time system** (`_mce_setup`, `frbus.py:334-363`): for a horizon of
   T quarters, duplicate every endogenous/exogenous variable and every
   equation T-1 times (`equations.dupe_vars` / `dupe_eqs`), producing one big
   system of `n_eqs x T` equations in `n_endos x T` unknowns.
3. **Lead removal / terminal condition** (`lexing.remove_leads`): a lead
   `v(+k)` in the period-t copy becomes the contemporaneous duplicate variable
   for period t+k; leads that run past the terminal period are substituted
   with fixed data values from the input frame (a "constant terminal
   condition").
4. **One Newton solve** over the stacked system with a sparse Jacobian
   (`jacobian.mce_create_jacobian`), plus block ordering/decomposition
   (`block_ordering.py`, `digraph_lib.py`) to keep the LU factorization
   tractable.
5. **Data plumbing** (`_populate_mce_data`, `data_lib.drop_mce_vars`,
   `copy_fwd_to_current`): the duplicated "lead columns" are populated before
   the solve and stripped from the returned frame.

## What our implementation already has

- Parsing of `mce_vars` and access to the raw XML (only the standard
  equations' text is currently extracted; an `mce_equation`/`mce_coeff`
  extractor is a small extension of `parse.py`).
- A lexer whose token form `(name, lag)` already represents leads (`rff(1)`
  lexes to `("rff", 1)`); only `xsub` assumes `lag <= 0`.
- A sparse-Jacobian Newton solver (`solver.newton`) that is agnostic to system
  size — the same code can solve a stacked system if given stacked `feqs`/
  `fjac`.
- Tracking-residual and exogenize machinery that operates on lists of lexed
  equations, which duplicates cleanly.

## Work plan (no architecture rewrite required)

The per-period machinery generalizes; the changes are additive:

1. **Parser** (~small): extract `<mce_equation>` text and `<mce_coeff>` values
   per variant; variant selection maps `mce` to the set of variables whose
   equations get swapped (mirror `CONST_MCE_TYPES`).
2. **Stacking module** (~the core work): given (equations, endo/exo names, T),
   emit duplicated names `var_1..var_{T-1}`, duplicated equations with
   time-shifted references, and terminal-condition substitution for leads
   beyond T. This is a pure text/lexed-token transformation, directly
   mirroring `dupe_vars`/`dupe_eqs`/`remove_leads`.
3. **`xsub` extension**: leads in the period-t equation copy resolve to the
   `x[...]` slot of the corresponding duplicate variable instead of `data`.
4. **Solve path**: an MCE `solve` builds the stacked system once per
   (start, end, layout), populates duplicate columns from the input frame,
   runs a single `newton`, and unpacks row 0..T-1 results back into the
   frame. `init_trac` under MCE is the same fixed-point evaluation as today.
5. **Performance guard**: a 284-equation model over 60 quarters is a
   ~17k-equation sparse system; the vendor needs block decomposition to make
   the LU affordable. Plan: start with plain sparse LU (correct first), and
   only add block ordering if solve times are unacceptable; scipy's `splu`
   with COLAMD ordering may be adequate for demo horizons (T <= 100).
6. **Validation**: gate against the vendor's `example2.py` (mce="mcap+wp",
   100bp monetary shock) exactly as the VAR scenarios are gated in
   tests/test_shock.py, with a committed reference generated by
   scripts/generate_vendor_reference.sh extended with an MCE scenario.

## Estimate and risks

- Effort: roughly 2-4 focused days; the stacking/lead-removal transformation
  and its tests dominate.
- Risks: (a) terminal-condition semantics must match the vendor exactly or
  cross-validation fails at the horizon edge; (b) Jacobian compile time for
  the stacked system (symbolic differentiation of ~17k equations) may need
  caching or exploiting the block-Toeplitz structure (each period's block is
  identical except at the boundaries — differentiate one block, reuse T
  times); (c) memory growth in `runtime.compile_equations` for large T.
- Not required: no changes to the Newton solver, data loading, stochsim, or
  the public API shape (`mce=` already exists in the constructor signature).
